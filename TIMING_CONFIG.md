## 📋 详细配置说明

### 1. 日志处理器（Log Processor）

**功能：** 从C2端拉取上线日志和清除日志

**配置文件：** `backend/config.py`

```python
# C2数据拉取间隔
C2_PULL_INTERVAL_MINUTES = 0.1667  # 0.1667分钟 = 10秒
```

**实际效果：**
- ✅ 每10秒拉取一次C2数据
- ✅ 超高实时性，适合测试和演示
- ⚠️ 会增加网络和数据库负载

**计算公式：**
```
10秒 ÷ 60 = 0.1667分钟
```

**在C2_ENDPOINTS中的应用：**
```python
C2_ENDPOINTS = [
    {
        'pull_interval': C2_PULL_INTERVAL_MINUTES * 60,  # 转换为秒 = 10秒
        # ...
    }
]
```

---

### 2. 统计聚合器（Stats Aggregator）

**功能：** 定期聚合统计数据（按小时、天、月）

**配置文件：** `backend/config.py`

```python
# 聚合间隔
AGGREGATOR_INTERVAL_MINUTES = 0.5  # 0.5分钟 = 30秒
```

**启动命令：** `start_all_v3.bat`

```batch
python aggregator.py daemon 0.5
```

**实际效果：**
- ✅ 每30秒执行一次聚合
- ✅ 快速更新统计数据
- ✅ 适合实时监控场景

**计算公式：**
```
30秒 ÷ 60 = 0.5分钟
```

## 🎯 性能影响分析

### 日志处理器（10秒间隔）

**优点：**
- ✅ 超高实时性，数据延迟最多10秒
- ✅ 适合演示和测试
- ✅ 用户体验好

**缺点：**
- ⚠️ 网络请求频繁（每分钟6次）
- ⚠️ 数据库写入频繁
- ⚠️ 适合小规模或测试环境

**建议：**
- 生产环境建议调整为1-5分钟
- 大规模部署建议5-10分钟

### 统计聚合器（30秒间隔）

**优点：**
- ✅ 统计数据更新及时
- ✅ 大屏显示响应快
- ✅ 对性能影响较小（主要是读操作）

**缺点：**
- ⚠️ 频繁的聚合计算
- ⚠️ 可能增加CPU使用率

**建议：**
- 测试环境：30秒（当前配置）
- 生产环境：5-10分钟

### 时间窗口重叠（防止边界数据丢失）
```python
# backend/config.py
C2_PULL_WINDOW_OVERLAP_RATIO = 0.4  # 40%重叠
C2_PULL_OVERLAP_MINUTES = max(0.1, C2_PULL_INTERVAL_MINUTES * 0.4)
```

**说明：**
- 拉取数据时会回溯一定时间
- 10秒间隔 × 40% = 4秒回溯

## 🔍 故障排查

### 问题1: 日志处理器不按10秒拉取

**检查：**
1. 确认 `backend/config.py` 中 `C2_PULL_INTERVAL_MINUTES = 0.1667`
2. 检查是否有环境变量覆盖：`echo %C2_PULL_INTERVAL_MINUTES%`
3. 查看日志确认实际间隔

### 问题2: 聚合器不按30秒聚合

**检查：**
1. 确认 `backend/config.py` 中 `AGGREGATOR_INTERVAL_MINUTES = 0.5`
2. 确认启动参数：`python aggregator.py daemon 0.5`
3. 查看进程是否正常运行
